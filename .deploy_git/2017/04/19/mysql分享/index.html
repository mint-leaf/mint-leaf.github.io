  • 作为最近在看的‘高性能 mysql‘的一个阶段性总结, 感谢这本书给我带来的巨大的收获

  • 关于怎么查看你所写的 sql 语句是否如你所愿：

    请在写 sql 时在前面加上explain，查看结果，如：explain select name from users where id = 2你就会的到一张表格，里面有关于你这条 sql 语句所有的信息。

  • 关于 mysql 的范式与反范式

    • 第一范式： 所有的 mysql 的属性都不可再分
    • 第二范式：在第一范式的基础上建立起来的， 要求数据库中的每个实例或者行必须唯一可分，通常通过建立一个主键来实现,但优化了一些第一范式的约束条件
    • 第三范式：表的信息， 能够被推到出来，也就是我们所使用的外键
    • 反范式： 全都放进一张表格里
    • 范式优点：
      • 更新操作快
      • 应对密集型应用的性能会很好
      • 基本没有数据冗余
      • 表通常比较小，（但现在好像不是这样了）
    • 范式的缺点：
      • 最大的缺点就是关联，常常需要关联多张表格
      • 如果多表连接查询时结果中只包含某张大表中很少的数据集时，不合理的表结构和 sql 语句会造成严重的性能问题
    • 反范式的优点：
      • 全都在一张表格里面，不用做关联
      • 易于删库跑路
    • 反范式缺点：
      • 数据全部在一张表格里面，会造成表格数据巨大
      • 会有很严重的安全问题和数据问题，而且有些涉及表锁的操作会产生巨大的影响
    • 混用范式和反范式：常见的做法是使用反范式建立缓存，使用范式化或者部分范式化的（在表格中适量的添加冗余数据），使用触发器更新缓存

  • 关于 mysql 的建表：

    混用范式和反范式，由于现在大多数由一张总表，数张子表，临时表构成， 所以在主表上尽可能的使用范式建表，尽可能的细化属性的颗粒大小，对于主表牵连的子表，在子表中添加一些冗余的数据列可以避免多表链接查询时牵连过多的数据列，这里添加的冗余的数据列尽量选择在代码中使用的较多的属性。将数据一致性的问题大部分交给代码逻辑进行处理 。临时表的构建使用反范式，可以更快的获得想要的结果，总之就是在多表链接，数据一致性，表格大小，易于维护性这四个方面进行均衡。

  • 关于 mysql 的索引：

    • 索引在 Innodb 的实现：因为使用的绝大多数是 Innodb 的引擎，所以只针对 Innodb 进行讨论，索引在 Innodb 中实现使用的 B-Tree 的结构， B-Tree 是一种平衡的完全树。对于 B-Tree 来讲，索引能够使用的最左前缀，什么是索引最左前缀？举个栗子，在 test 的表里面有一个叫 combine 的索引，，这个索引由两个列构成（id, name）其中 id 是主键，那么select id, name from test where name = 'nb'就不能使用到这个索引，这就是索引最左前缀的意思。再看下面的这个栗子：假设建有如下的表：
      CREATE TABLE people (
      last_name varchar(50) not null,
      first_name varchar(50) not null,
      dob date not null,
      sex enum(‘man’, ‘woman’, ‘other’) not null,
      key (last_name, first_name)
      )
      这个索引用 B-Tree 表示就是这样的:。
      note：索引对多个值进行排序是按照建表时候定义索引的列顺序来排序的
    • 高性能索引的实现：
      • 独立的列：独立的列的意思是索引列不能是表达式的一部分，再举个栗子： 如果在id上有个索引，那么在where中这样写的话where id + 1 = 5那么就不能使用到这个索引，
      • 对于比较长的列，比如类似于name VARCHAR（255）这样的，可以使用name(12)这样来建立索引列，或者再进行一次哈希，使得索引的区分度更高，什么索引的区分度，就是在指定范围内索引的重复度，比如在某个表上有一个year字段，而这个字段里的数据就两种2015，2016，那么如果数据量较大，在这上面建立索引并更根据这个索引查询是不明智的，可以建立一个索引KEY(month, year, day)来提高区分度，所以，对于查询来讲，在做where的条件过滤时，尽可能将区分度高的条件放在前面，
      • 索引覆盖查询：索引覆盖查询的意思就是你所要的结果列就是一个索引，那么这个查询的速度就能得到极快的提升，但是，这个及其考验整张表的结构设计和查询的写法，在explain所展示的框图中，索引覆盖查询的特征就是Extra：Using Index，但要注意的就是，对于返回的是很大的结果集的话，看不到索引的优化效果
      • 对于索引做排序：当做联结查询需要用到排序的时候，最好的结果就是排序的关键字在第一张表里面并且做排序的列是索引列。

  • 关于 Mysql 的查询优化：

    • 多表查询的坑.对于这个查询：
      select _ from actor inner join film_actor using (actor_id)
      inner join film using (film_id)
      where film.tittle = “mike”
      会返回多少数据列？答案是所有，三张表的所有数据列，因此，应该写成select actor._
    • 查询到底做了什么：
    • 查询优化器：查询优化器是很重要的一部分。查询优化器会调整你多表查询的表的关联顺序，查询优化器和衡量你多表查询的代价从而得到最优的结果，但是如果对于很多的表的联结，就会得到可能不是很好的结果（比如十张表），因为这时候优化器会采用贪心算法进行表的顺序的组合，因此得到的可能不是最好的结果。因此只要不是关联很多的表，就可以了
    • 那么对于很多表的查询，该怎么办：第一个方法，就是重组表的结构，因为出现 6 张以上表的查询就已经很不可思议了，所以对于很多的数据限制表，可以考虑用enum来进行代替，第二个方法，就是合理的进行索引，这个在索引那里讲过了，第三个，就是编写合理的查询，其中的一点就是，在之前版本的 mysql 中，对于子查询的优化是很糟糕的，因此在之前版本的 mysql 中，就是推荐用关联查询代替子查询，而且，在做子查询过程中通常会建立子表或者临时表，因为子表或者临时表上是没有索引的，如果子表或者临时表很大的化，就会产生问题，最应该做的，就是在ON或者USING的列上使用索引，一般来说，除非有特定的理由，否则只需要在关联顺序的第二张表上相应的列上建立索引就够了，同时，如果涉及到ORDDER BY或者GROUP BY操作，那么，应该确保那些操作里面只涉及一个表的列。
    • 对于 limit 优化。在很多操作里面，需要我们做分页(那个 ORM 的分页可以不用管，但是像之前写的关联 10 张表的还要做分页就真的是阿西吧了)，一个很头大的问题就是偏移量很大时候，这时候可以使用预知上界进行推测的方法，举个栗子，比如你的查询里面又有一个limit 2000, 4那么，如果你知道这个查询是在多少范围内的数据集，那么你就可以使用BETWEEN AND来重写这个查询，也就是我们使用一个书签来标记我们数据集的范围

  • 写在最后：

    mysql的优化方式有很多，上面提到的哪怕是索引那部分，在整个mysql索引那一块里面也只是冰山一角，唯一的方法就是多看书，然后把自己写的每一条sql语句进行仔细的推敲，才能达到比较好的结果和自己的提升。
