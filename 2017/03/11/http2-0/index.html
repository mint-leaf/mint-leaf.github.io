知乎live传送门

http2协议里的一些概念:

  • Stream Multiplexing:

    在一个http2的connection里， stream multiplexing是保证对并发的支持， 每个request被处理成 一个stream， 每个stream分配一个unique ID，保证并发能正确地被支持.

  • Server Push:

    server端能够主动地向客户端发起一个push，发送files或者一个new stream，考虑如下的场景：
    request->server, server->[html files, css files, js files, pngs…]->client
    这些css files和js files以及pngs必须在html被解析时或者被解析后才知道，然后去向server端
    请求资源，但由于server端在发送html时知道client需要哪些css files和js files以及pngs，所以
    server端可以push过去而不需要client发送额外的请求，所以能够提升页面加载速度

  • Binary Format and Header Compression:

    在http1.1及其之前的版本中，所有的请求带上的headers都是以文本形式发送出去的，占用资源比较高
    而http2.0支持对headers进行压缩， 压缩的格式可以使HPACK及其他

  • Flow control and Stream Prioritization:

    在http2.0之前， 每个request及response的内容都是以文本形式发送的，但对于http2.0来讲，
    每个request和response的traffic都会被打包成一个个的frame，每个frame都有自己的header和
    payload，以binary的形式发送，每个frame的header里都有包含这个stream的unique ID，
    从另一侧来的frame根据它的unique ID被分配到不同的stream里面

    这里面flow control的意思是接受队列缓冲区的大小通过一个frame来实现的，称之为
    window_update_frame，
    这里面stream prioritization的意思是定义了同一个connection下不同stream之间的依赖关系
    用作一种提醒机制，提示应该先接收哪一个，不过可以选择忽略

对于其中的一些解释：

  • 为何quic能够实现reliable：

    这是因为quic在传输层运用的UDP，而在应用层使用了TCP里的一些东西来帮助其实现reliable

  • 什么是Connection Migration，为什么http2.0不支持Connection Migration：

    Connection Migration简而言之就是使用wifi时，同一个SSD但有可能IP不一样。
    因为http2.0是基于TCP的IP+PORT， 所以对于这个是不支持的，而quic不一样， 因为quic是基于UDP的使用的是
    UUID来标记connection而不是IP+PORT

http2.0目前存在的一些问题：

  • case 1：
    因为http2.0是推荐使用一个connection，所以会发生HOLB的情况，考虑下面的情形:你有一个大文件正在传输，几乎
    占满了你server端的整个buffer，但是又有小文件要过来，这就只能把小文件安排在大文件之后，而对此，htt1.1的做法是
    重新开一个connection，但对于http2.0来讲，由于只推荐你使用一个connection，所以会有这样的结果，而且，对于client
    来讲，由于一开始并不知道文件的大小，所以并不会分成几个connection

  • case 2:
    大文件下载太慢，想要cut掉，对于http1.1来讲，浏览器直接cut掉了那个connection，但对于http2.0来讲，由于
    在server端的sending buffer里面还有大文件剩下的bytes，这些都会被发送，造成流量的浪费
